# CLAUDE.md

行为准则，旨在减少常见的 LLM 编码错误。可根据项目特定指令进行合并。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考，再编码

**不要假设。不要掩盖困惑。主动展示权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种理解，全部列出来——不要默默地选一个。
- 如果有更简单的方案，说出来。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑之处。问清楚。

## 2. 简单优先

**用最少的代码解决问题。不做任何推测性设计。**

- 不添加未被要求的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的情况编写错误处理。
- 如果你写了 200 行代码，而 50 行就能搞定，那就重写。

问问自己："一个资深工程师会说这太复杂了吗？" 如果是，就简化。

## 3. 精准修改

**只改必须改的。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的代码。
- 遵循现有风格，即使你会用不同的方式。
- 如果你发现无关的死代码，提一下——不要删除它。

当你的改动产生了孤立代码：
- 删除因你的改动而变得未使用的 import/变量/函数。
- 不要删除之前就存在的死代码，除非被要求。

检验标准：每一行改动都应该能追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写一个能重现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，先简要说明计划：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

强有力的成功标准让你可以独立循环。薄弱的标准（"让它能跑就行"）则需要不断澄清。

---

**这些准则生效的标志：** diff 中不必要的改动更少，因过度复杂导致的重写更少，澄清问题出现在实现之前而非犯错之后。

## 5. 分支策略

**每次执行新任务开发、修改代码时，必须从 main 创建新分支，禁止直接在 main 上开发。**

详细策略参见 [CONTRIBUTING.md](CONTRIBUTING.md)。
